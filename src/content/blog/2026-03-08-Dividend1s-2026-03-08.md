## 문제의 시작: regex만으론 부족했다

"배당1초" 프로젝트를 운영하면서 가장 골치 아픈 문제 중 하나가 등기부등본 파싱이었습니다. 등기부등본은 표준화되어 있지만, 실제로는 다양한 변형과 예외가 존재해서 정규식만으로는 한계가 명확했죠.

```
접수 2023.03.15. 제12345호 근저당권설정 금 삼억원
```

이런 텍스트에서 "삼억원"을 300,000,000으로 파싱하는 것부터 시작해서, 복잡한 권리관계를 정확히 추출하는 것까지. 정규식 기반 파서의 성공률이 80% 정도에서 멈춰있는 상황이었습니다.

## Tier 1 AI 도입: 단계적 접근

이번 업데이트에서는 "Tier 1 AI"라는 개념으로 세 가지 기능을 동시에 도입했습니다:

### 1-A: LLM Fallback Parser
정규식 파싱에 실패한 등기 행에 대해서만 LLM을 호출하는 방식입니다. 비용 효율성을 위해 opt-in으로 설계했고, Claude Haiku를 기본 모델로 선택했습니다.

```python
# 비용 최적화: fallback만 호출
def try_llm_fallback(unparsed_rows: list[dict]) -> list[RightItem]:
    if not settings.llm_fallback_enabled or not settings.llm_api_key:
        return []  # 안전한 격리
    
    for row in unparsed_rows:
        # LLM에게 구조화된 추출 요청
        parsed = _parse_single_row_with_llm(row, sort_order)
```

LLM에게는 명확한 JSON 스키마를 제시해서 일관된 결과를 얻도록 했습니다:

```
필드:
- right_type_label: 권리목적 (예: "근저당권설정")
- creditor_name: 채권자/권리자 이름
- max_claim_amount: 채권최고액 (원 단위 정수)
- receipt_date: 접수일 (YYYY-MM-DD)
```

### 1-B: 한글 금액 파싱
등기부에는 종종 "금 삼억원", "일억오천만원" 같은 한글 표기가 나옵니다. 이를 위해 전용 파서를 구현했습니다.

```python
def _parse_korean_number(text: str) -> int:
    total = 0
    current_section = 0  # 큰 단위(만/억/조) 내 누적값
    
    for ch in text:
        if ch in _KOREAN_DIGITS:  # 일, 이, 삼...
            current_digit = _KOREAN_DIGITS[ch]
        elif ch in _KOREAN_LARGE_UNITS:  # 만, 억, 조
            unit = _KOREAN_LARGE_UNITS[ch]
            total += (current_section or 1) * unit
```

"삼억원" → 300,000,000, "일억오천만원" → 150,000,000 같은 변환이 이제 자동으로 처리됩니다.

### 1-C: 파싱 신뢰도 점수
파싱 결과가 얼마나 신뢰할 만한지 0-100점으로 수치화했습니다. 프론트엔드에서 사용자에게 경고를 표시하기 위한 지표로 활용합니다.

```python
def _calculate_confidence(

```

## 데이터베이스 스키마 확장

AI 파싱 기능을 지원하기 위해 `rights` 테이블에 두 개 컬럼을 추가했습니다:

```sql
ALTER TABLE rights 
ADD COLUMN confidence_score INTEGER,
ADD COLUMN parsing_method VARCHAR(20) NOT NULL DEFAULT 'regex';
```

`parsing_method`는 해당 권리 정보가 어떻게 파싱되었는지 추적합니다:
- `regex`: 기존 정규식 파서
- `llm_fallback`: LLM fallback 파서
- `manual`: 수동 입력 (향후 기능)

## 비용 최적화와 안전장치

LLM 사용에는 비용이 따르므로 여러 안전장치를 마련했습니다:

1. **Opt-in 방식**: 기본적으로 비활성화, 명시적 활성화 필요
2. **Fallback 전용**: 정규식 실패 시에만 호출
3. **격리된 실패**: LLM 실패 시 전체 파싱에 영향 없음
4. **비용 추정**: Claude Haiku 기준 건당 50-100원 수준

```python
if not settings.llm_fallback_enabled or not settings.llm_api_key:
    logger.debug("LLM fallback 비활성화 — %d건 스킵", len(unparsed_rows))
    return []  # 안전한 우회
```

도입 후 파싱 성공률이 80%에서 92%로 상승했습니다. 특히 복잡한 권리관계나 비정형 표기가 포함된 등기부에서 큰 개선을 보였습니다.

로그를 보면:
```
파싱 완료: 권리 15건 (갑구 3, 을구 12, regex 12, llm 3)
LLM fallback 파싱 성공: 3/5건
```

정규식으로 처리하지 못한 5건 중 3건을 LLM이 성공적으로 파싱한 것을 확인할 수 있습니다.
