## 왜 리브랜딩이었나

사주 MBTI 리포트는 처음에 '한지·먹·봉인 인장·마법서(그리무아)' 같은 동양 판타지 어휘로 앱 전체를 감쌌다. 분위기는 있었지만 문제가 쌓였다. 폰트만 4종(Black Han Sans, Nanum Myeongjo, Gowun Batang, Pretendard)이라 모바일 초기 로드가 느렸고, "마음 문을 여는 중…" 같은 카피는 사용자에게 낯설었다. 결정적으로 캐릭터 디자인이 산리오·이모티콘 계열의 말랑하고 생활감 있는 방향으로 굳어지면서, UI와 캐릭터의 감성이 완전히 따로 놀기 시작했다.

리브랜딩 v2의 방향은 단순했다. 캐릭터가 주인공이 되도록 UI는 더 희고, 더 비우고, 더 둥글게. 카피는 세계관 설명 대신 한 문장짜리 생활 감정으로.

## Phase 1 & 2: 토큰과 컴포넌트부터

리스크를 줄이기 위해 단계를 나눴다. 가장 먼저 한 건 **CSS 토큰 값만 갈아치우기**였다. 컴포넌트 클래스명은 그대로 두고 변수가 가리키는 값만 바꾸면, 21개 파일을 손대지 않고도 전체 팔레트가 한 번에 전환된다.

```css
/* 이전 */
--color-parchment: #fbf4e4;
--color-ink: #1c1b2a;
--color-gold: #c28f2e;

/* 이후 */
--color-bg: #fffdf8;
--color-line: #4a443e; /* 웜 먹색 — 캐릭터 선과 동일 계열 */
--color-butter: #f7d98c; /* 주 CTA */
--color-cheek: #f4a9b0; /* 브랜드 포인트 */
```

폰트는 4종에서 2종으로 줄였다. 명조·붓글씨 계열을 폐기하고 **Jua(주아)** 단일 디스플레이 폰트로 교체했다. Jua는 400 웨이트만 있어서 `.t-title`, `.t-punch` 클래스의 font-weight도 800/700에서 400으로 내렸다. 합성 볼드 렌더링 방지를 위한 결정이다.

Phase 2에서는 컴포넌트 이름을 정리했다. `ParchmentCard → SoftCard`, `GoldButton → PrimaryButton`, `WaxButton → AccentButton`, `ChapterHeading → SectionTitle`. 한 번에 rename하지 않고 신규 이름을 만들고 구 이름을 deprecated alias로 유지했다. 기존 import를 쓰는 사용처가 많았기 때문에, 한꺼번에 교체하면 리뷰 범위가 너무 커진다.

## Phase 3: 카피 전면 교체

코드보다 손이 더 많이 간 작업이었다. 앱 전역에 흩어진 카피를 하나씩 찾아 교체했다.

- "마음 문을 여는 중…" → "사주몬을 만드는 중…"
- "사주몬 깨우기" → **"사주몬 데려오기"** (전역 40여 곳)
- "이웃" → "친구", "마을로 돌아가기" → "처음으로"
- "N장 깨어남" → "N장 열림"
- "다시 불 밝히기" → "다시 시도"

세계관 안 B(사주몬 동네)를 선택한 이유는 도감·수집 루프 때문이었다. "아직 만나지 못한 이웃"이나 "옆집" 같은 공간 은유가 수집 메커니즘과 자연스럽게 연결됐다. 단, 마법서·봉인·의식 같은 판타지 어휘만 제거하고 공간감 자체는 경량으로 유지했다.

## Phase 4A: MBTI 입력 구조 뒤집기

MBTI를 모르는 사람을 위한 10문항 퀴즈가 기본 경로였다. MBTI를 이미 아는 사람이 대부분인데 전원에게 퀴즈를 강제하는 구조였다.

```tsx
// 이전: 퀴즈가 주 경로
<button onClick={() => setPhase('quiz')}>내 성향 찾기 (1분)</button>

// 이후: 16 코드 그리드가 주 경로, 퀴즈는 폴백
<div className="grid grid-cols-4 gap-1.5">
  {MBTI_CODES.map((code) => (
    <button key={code} onClick={() => pickDirect(code)}>{code}</button>
  ))}
</div>
<button onClick={() => setPhase('quiz')}>모르겠어요 · 성향 찾기 (1분)</button>
```

퀴즈 문항 10개도 전면 재작성했다. "새벽빛이 스미자 알 껍데기에 금이 갑니다" → "새 모임에 처음 나간 날. 아는 얼굴은 없고 사람들은 이미 이야기 중이에요." 이런 식으로 판타지 장면을 실제 생활 장면으로 교체했다. 중요한 건 각 선택지에 매핑된 `poles`(축 정보)는 한 글자도 건드리지 않아서 `scoreMbti` 진단 결과가 완전히 보존됐다는 점이다.

## Phase 4B·C와 레이아웃 재설계

공유 버튼이 리포트 맨 아래에 있었다. 15장 문서를 끝까지 스크롤해야 닿는 동선이라, 도감·수집 루프의 시작점이 사실상 닫혀 있었다. 유료 리포트에 한해 히어로 카드 직후로 공유를 끌어올렸다. 캐릭터를 막 본 시점이 공유 동기가 가장 크기 때문이다.

랜딩 레이아웃도 뒤집었다. 전체 화면을 차지하던 일러스트가 첫 화면에 있어서, 입력 폼을 보려면 스크롤이 필요했다. **BrandHeader**를 신설해 로고와 태그라인을 100px 안에 압축하고 입력 폼을 첫 화면에 올렸다. 아트워크는 입력 아래 분위기 전달용으로 역할을 바꿨다.

리포트 뷰는 15장 세로 아코디언에서 마당 단위 가로 덱으로 교체했다. **ReportDeck**은 CSS `scroll-snap`만으로 스와이프를 구현했다. 제스처 라이브러리를 추가하지 않은 건 의도적 선택이다. 마당 안 장들은 세로로 읽고, 마당 간 이동은 가로로 스와이프한다. 두 축이 달라서 제스처가 겹치지 않는다.

무료 리포트 화면에서는 결제 유도 블록이 4개였다(요약 카드 마지막 줄 → 진행 카드 → 블러 프리뷰 → CTA). 같은 메시지를 네 번 반복하는 구조를 하나의 카드로 합쳤다. 진행바 + 잠긴 마당 미리보기 + CTA를 한 블록에. 블록 12개 → 9개, 결제 유도 4개 → 1개.

## 결과

당일 오전 스펙 확정, 당일 오후 6개 PR 머지. 컴포넌트 테스트 73개에서 486개로 늘었고(덱 신규 5개 포함) `next build`와 `tsc`는 모든 단계에서 클린을 유지했다. 각 Phase를 독립 배포 가능하게 설계한 덕분에 중간 상태도 프로덕션에 올릴 수 있었다.

아직 남은 건 LLM 프롬프트 보이스(Phase 5)와 캐릭터 에셋 v2 교체(Phase 6)다. 특히 프롬프트 전면 교체는 캐시 전량 무효화와 첫 생성 비용 급증을 동반하기 때문에, baseline 폴백률을 1주간 확보한 뒤 진행할 예정이다.
