## 프로젝트 소개

최근 "배당1초"라는 흥미로운 프로젝트를 개발하고 있습니다. 부동산 등기부등본 PDF를 업로드하면 NPL(부실채권) 배당금과 수익률을 1초 만에 계산해 리포트로 제공하는 B2B/B2H 웹 서비스입니다. 이번 포스트에서는 핵심 백엔드 시스템 구축 과정을 공유해보겠습니다.

## 아키텍처 설계 철학

전체 시스템은 마이크로서비스 아키텍처로 설계했습니다. 각 서비스가 명확한 책임을 가지도록 분리한 이유는 다음과 같습니다:

- **파싱 서비스**: PDF 처리와 OCR은 Python 생태계가 압도적으로 강력합니다
- **코어 API**: 비즈니스 로직과 결제는 Java/Spring의 안정성이 필요합니다
- **확장성**: 각 서비스를 독립적으로 스케일링할 수 있습니다

## FastAPI 파싱 서비스 구축

### PDF 텍스트 추출 전략

파싱 서비스의 핵심은 등기부등본 PDF에서 정확한 텍스트를 추출하는 것입니다. 이를 위해 2단계 폴백 전략을 구현했습니다:

```python
async def extract_text(pdf_bytes: bytes) -> str:
    """PDF 바이트에서 텍스트 추출. 텍스트가 부족하면 OCR 폴백 실행."""
    text = _extract_with_pdfplumber(pdf_bytes)
    
    if len(text.strip()) < OCR_FALLBACK_THRESHOLD:
        logger.info(
            "추출 텍스트 %d자 — OCR 폴백 실행 (임계값: %d자)",
            len(text.strip()),
            OCR_FALLBACK_THRESHOLD,
        )
        from .ocr_service import extract_with_ocr
        text = await extract_with_ocr(pdf_bytes)
    
    return text
```

1차로 `pdfplumber`로 텍스트 추출을 시도하고, 결과가 500자 미만이면 이미지 PDF로 판단하여 OCR을 실행합니다. OCR도 Tesseract 로컬 처리 후 Google Vision으로 폴백하는 이중 안전망을 구축했습니다.

### 등기부 파싱 로직

등기부등본은 갑구(소유권)와 을구(소유권 외 권리)로 나뉩니다. 각 섹션에서 권리 관계를 추출하여 구조화된 데이터로 변환하는 것이 핵심입니다:

```python
def parse_registry(text: str) -> ParsedRegistry:
    """등기부 전문 텍스트를 파싱해 ParsedRegistry 반환."""
    result = ParsedRegistry()
    result.property_address = _extract_property_address(text)
    result.property_details = _extract_property_details(text)

    section_a_text, section_b_text = _split_sections(text)
    result.section_a_raw = _extract_raw_rows(section_a_text)
    result.section_b_raw = _extract_raw_rows(section_b_text)

    # 각 행을 RightItem 객체로 변환
    rights = []
    sort_order = 1
    
    for row in result.section_a_raw:
        right = _row_to_right_item(row, "A", sort_order)
        if right:
            rights.append(right)
            sort_order += 1
    
    result.rights = rights
    return result
```

등기부에는 "근저당권설정", "저당권말소" 같은 다양한 표현이 나타납니다. 이를 시스템에서 일관되게 처리하기 위해 정규화 매핑을 구현했습니다:

```python
RIGHT_TYPE_MAP: dict[str, str] = {
```

이런 식으로 원문 표현을 표준 코드로 매핑하여 후속 배당 계산 로직에서 일관되게 처리할 수 있게 했습니다.

## Spring Boot 코어 API 설계

### JPA 엔티티 설계

총 10개의 테이블로 구성된 데이터 모델을 설계했습니다. 핵심은 `AnalysisRequest`를 중심으로 한 상태 머신 패턴입니다:

```java
@Entity
@Table(name = "analysis_requests")
public class AnalysisRequest extends BaseEntity {
    
    @Enumerated(EnumType.STRING)
    @Builder.Default
    private AnalysisStatus status = AnalysisStatus.PENDING;
    
    // 상태 전환 메서드
    public void startParsing() {
        this.status = AnalysisStatus.PARSING;
    }
    
    public void startCalculating() {
        this.status = AnalysisStatus.CALCULATING;
    }
    
    public void complete() {
        this.status = AnalysisStatus.COMPLETED;
    }
    
    public void fail(String errorMessage) {
        this.status = AnalysisStatus.FAILED;
        this.errorMessage = errorMessage;
    }
}
```

### JSONB 활용한 유연한 데이터 저장

권리 정보는 구조가 복잡하고 원문 보존이 중요하므로 PostgreSQL의 JSONB를 적극 활용했습니다:

```java
@Entity
@Table(name = "rights")
public class Right {
    
    @Type(JsonBinaryType.class)
    @Column(name = "raw_data", columnDefinition = "jsonb")
    private Map<String, Object> rawData = new HashMap<>();
    
    // 구조화된 필드들
    private String rightType;
    private String creditorName;
    private String debtorName;
    private Long maxClaimAmount;
    // ...
}
```

이렇게 하면 파싱된 원문 데이터를 완전히 보존하면서도, 쿼리와 계산에 필요한 핵심 필드들은 정규화된 컬럼으로 저장할 수 있습니다.

### 서비스 간 통신

코어 API에서 파싱 서비스를 호출하기 위해 Spring의 새로운 `RestClient`를 사용했습니다:

```java
@Component
public class ParsingServiceClient {
    
    private final RestClient restClient;
    
    public ParsingServiceClient(RestClient.Builder builder) {
        this.restClient = builder
            .baseUrl(parsingServiceUrl)
            .defaultHeader("X-Internal-Token", serviceToken)
            .build();
    }
    
    public ParseRegistryResponse parseRegistry(MultipartFile file, UUID analysisId) {
        return restClient.post()
            .uri("/parse/registry")
            .body(createMultipartBody(file, analysisId))
            .retrieve()
            .body(ParseRegistryResponse.class);
    }
}
```

내부 서비스 간 통신은 `X-Internal-Token` 헤더로 인증하도록 설계했습니다.

## 보안과 인증

### JWT 기반 인증

Stateless 아키텍처를 위해 JWT 토큰 기반 인증을 구현했습니다:

```java
@Component
public class JwtTokenProvider {
    
    public String generateToken(String userId) {
        return Jwts.builder()
            .subject(userId)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(secretKey)
            .compact();
    }
    
    public boolean validateToken(String token) {
        try {
            Jwts.parser().verifyWith(secretKey).build().parseSignedClaims(token);
            return true;
        } catch (JwtException | IllegalArgumentException e) {
            return false;
        }
    }
}
```

### CORS 및 보안 설정

마이크로서비스 환경에서 적절한 CORS와 보안 설정이 중요합니다:

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.sessionManagement(session -> 
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/v1/auth/**", "/api/v1/payments/webhook/**").permitAll()
                .requestMatchers("/health", "/docs", "/swagger-ui/**").permitAll()
                .anyRequest().authenticated())
            .addFilterBefore(jwtAuthenticationFilter, 
                UsernamePasswordAuthenticationFilter.class);
        
        return http.build();
    }
}
```

## 결제 시스템 통합

Paywall 모델을 구현하기 위해 Toss Payments를 통합했습니다:

```java
@Entity
@Table(name = "payments")
public class Payment {
    
    @Enumerated(EnumType.STRING)
    private PaymentStatus status = PaymentStatus.PENDING;
    
    @Type(JsonBinaryType.class)
    @Column(name = "raw_response", columnDefinition = "jsonb")
    private Map<String, Object> rawResponse = new HashMap<>();
    
    // 결제 상태 전환
    public void complete(String paymentKey, Map<String, Object> response) {
        this.status = PaymentStatus.COMPLETED;
        this.paymentKey = paymentKey;
        this.rawResponse = response;
        this.paidAt = OffsetDateTime.now();
    }
}
```

## 마무리

이번에 구축한 시스템의 핵심은 **관심사의 명확한 분리**입니다. PDF 파싱은 Python이, 비즈니스 로직은 Java가 각각 최적의 성능을 발휘할 수 있도록 설계했습니다. 

마이크로서비스 아키텍처의 복잡도는 증가했지만, 각 서비스의 단일 책임 원칙을 지키면서 확장 가능한 시스템을 만들 수 있었습니다. 특히 JSONB를 활용한 유연한 데이터 저장과 상태 머신 패턴을 통한 프로세스 관리가 핵심 포인트였습니다.
